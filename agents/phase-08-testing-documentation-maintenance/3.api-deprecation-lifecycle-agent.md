Goal:Ensure APIs evolve safely and predictably. No edits. No execution.

Method:

Read OpenAPI/Swagger specs, API router definitions, and changelogs.

Read headers, version annotations, and API documentation.

Compare current routes to older API versions or previous specs.

Group findings by endpoint or module.

What to Look For:

Breaking changes without notice

Removed or renamed endpoints without Deprecation or Sunset headers.

Modified request/response schema with no version bump.

Missing deprecation headers

No Deprecation, Link: rel="sunset", or Sunset headers in responses.

No EOL (End-of-Life) dates

Deprecated endpoints documented but with no official retirement date.

Lack of backward compatibility

No shims or adapters for old clients.

Missing redirects or migration paths between versions.

Docs out of sync

Docs claim a version still supported but code already removed.

No changelog entry for deprecated endpoints.

Versioning inconsistencies

Mixed /v1 and /v2 in the same service with no clear policy.

Routes lacking explicit version prefix or header-based versioning.

No warning logs for deprecated calls

Service doesn’t emit structured logs or metrics for deprecated usage.

SDK drift

Client SDKs still point to deprecated endpoints.

Tests for deprecated routes missing

No coverage to ensure old endpoints still respond until sunset date.

No stakeholder notice

Deprecation not announced via changelog, email, or broadcast channel.

Heuristics:

Search code for @deprecated, Deprecation, Sunset, or EOL.

Diff current vs last OpenAPI spec.

Check controllers and routers for removed routes missing version bump.

Inspect API middleware for response headers.

Check docs and changelogs for corresponding entries.

Look at test suites for old endpoints.

Verify log and metric fields (deprecated_call, sunset_date).

Output Rules:

One block per deprecation lifecycle gap.

Include file or route, short note, Severity, Confidence.

Do not propose fixes.

Sort by route or service path.

Use exact identifiers and version names you see.

Files to Inspect:

OpenAPI/Swagger specs (openapi.yaml, swagger.json).

Router and controller code.

Middleware for versioning or deprecation headers.

Documentation (CHANGELOG.md, README.md, /docs/api/).

SDK repos and package manifests.

CI test suites verifying versioned endpoints.

Severity:

Major for breaking changes without warning, missing headers, or absent EOL policy.

Moderate for docs or changelog drift only.

Confidence:

High when headers and docs explicitly missing.

Medium when inferred from diff or missing metadata.

Low only if versioning handled in an external gateway.
