Goal:Minimize app permissions and data collection surface. No edits. No execution.

Method:

Read Android AndroidManifest.xml, iOS Info.plist, and any platform-specific permission files.

Read privacy policy metadata (App Store, Play Store, GDPR disclosures).

Compare declared permissions to code usage and documented purpose.

Group findings by platform and manifest file.

What to Look For:

Unused permissions

Permissions declared but never used in code (READ_CONTACTS, ACCESS_FINE_LOCATION, etc.).

Permissions left over from old SDKs or deprecated features.

Overly broad permissions

Full storage or contact access when partial access APIs exist.

READ_EXTERNAL_STORAGE instead of scoped access.

Background data collection

Background location tracking, analytics, or sensors running without user consent toggle.

Missing privacy strings

iOS apps missing NSCameraUsageDescription, NSLocationWhenInUseUsageDescription, etc.

Privacy statements generic or blank.

No runtime permission prompts

Android apps requesting permissions at install instead of runtime.

SDK or library drift

Third-party SDKs adding sensitive permissions (ads, analytics, trackers).

Platform misalignment

iOS capabilities in Xcode not matching manifest.

Android feature tags missing required="false" for optional hardware.

No opt-out or data deletion hooks

Users can’t revoke tracking or delete data within the app.

Analytics / telemetry untagged

No clear consent flow before telemetry collection begins.

Missing app transport security (ATS)

iOS NSAppTransportSecurity allows insecure HTTP.

Bluetooth or NFC without usage reason

Permissions declared but no feature referencing them.

Push notifications with no category opt-outs

Notifications on by default, no user settings for topics.

Permissions used by background services

Services request location or sensors even when app closed.

Desktop equivalents (Electron, PWA)

No permission gating for filesystem or clipboard APIs.

Heuristics:

Android: search uses-permission tags, cross-check with code references.

iOS: read Info.plist and Swift/Obj-C usage of permission APIs.

Look for permissions never referenced in source or only in imported SDKs.

Review NSAppTransportSecurity for NSAllowsArbitraryLoads=true.

Identify ACCESS_BACKGROUND_LOCATION, BLUETOOTH_ADMIN, RECORD_AUDIO, READ_CONTACTS.

Compare telemetry libraries to declared consent flow.

Output Rules:

One block per permission or feature gap.

Include file, platform, short note, Severity, Confidence.

Do not suggest fixes.

Sort by platform, then file path.

Use exact permission or key names as found in the manifest.

Files to Inspect:

AndroidManifest.xml, AndroidManifest_debug.xml

iOS Info.plist, Entitlements.plist, and project capabilities

Electron or desktop app configs

Store metadata (privacy-manifest.json, app-store-metadata.yaml)

SDK dependency manifests

Severity:

Major for unused or overbroad permissions, missing privacy strings, background tracking.

Moderate for incomplete opt-outs or insecure ATS configs.

Critical for permissions that can expose sensitive user data (camera, mic, contacts) without notice.

Confidence:

High when explicit manifest fields or SDK permissions are visible.

Medium when inferred from code usage or partial configs.

Low only if permissions dynamically added at runtime.
