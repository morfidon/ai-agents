Goal:Ensure the system can actually recover after a failure. No edits. No execution.

Method:

Read infrastructure templates (Terraform, Helm, CloudFormation).

Read backup scripts, snapshot schedules, and runbooks.

Read cron definitions or CI jobs related to backups.

Compare backup definitions with restore procedures.

Group by system or data store.

What to Look For:

No backup configuration

Databases or storage missing scheduled backups.

Cloud resources without snapshot policies.

Missing restore scripts

No automation to restore data from backups.

No defined RTO (recovery time objective).

Untested backups

No record or pipeline for restore drills.

Backups taken but never validated.

Snapshots incomplete or expired

Old snapshots deleted with no retention rotation.

Newer resources not covered by existing plan.

Encryption key mismatch

Backups encrypted with keys not rotated or now revoked.

Missing mapping between backup and KMS key alias.

Cross-region / cross-zone gaps

Backups stored in same region as primary.

No multi-region copy for disaster recovery.

Configuration data not included

Only DB snapshots, no config or secrets backup.

Missing .env or parameter store dumps.

Immutable backup policy missing

No write-once, read-many (WORM) setup for critical data.

Retention misalignment

Business policy requires 90 days; IaC defines 7.

Logs retained shorter than compliance period.

Dependency mismatch

Apps reference restored DBs with old schema or credentials.

Storage drift

Backup bucket not versioned, not encrypted, or publicly accessible.

No alerting on failed backups

No health check or alarm when jobs fail.

Manual steps undocumented

Restores rely on tribal knowledge, not runbooks.

CI/CD restores missing

Staging restores not part of pipelines.

DR (Disaster Recovery) drills absent

No simulated recovery scenarios logged in past 6 months.

Heuristics:

Terraform: aws_backup_plan, aws_backup_vault, aws_db_instance.backup_retention_period, google_sql_database_instance.settings.backup_configuration.

Helm/K8s: CronJob for snapshots, sidecar scripts, Velero manifests.

Logs: look for “restore test”, “snapshot verification”, “rto test”.

Policies: check retention_in_days, lifecycle_rules, encryption flags.

Key rotation: match backup KMS key to alias; flag missing rotation schedule.

Restore: search for “restore.sh”, “restore.yaml”, “recovery_playbook.md”.

Output Rules:

One block per gap.

Include system or component, short note, Severity, Confidence.

No fixes or commands.

Sort by system or data store.

Use exact identifiers and resource names from the code.

Files to Inspect:

Terraform/CloudFormation templates.

Helm charts and K8s CronJobs.

Backup scripts (backup.sh, snapshot.py).

Restore docs and playbooks.

KMS and retention configs.

CI/CD pipeline YAMLs.

Runbooks in /ops, /infra, /docs.

Severity:

Major for missing backups, no restores, no encryption, or single-region storage.

Moderate for incomplete retention or missing drills only.

Minor if all backups exist but policy docs outdated.

Confidence:

High when explicit fields are absent or false.

Medium when inferred from missing scripts or logs.

Low when policy sources are incomplete or environment unclear.
