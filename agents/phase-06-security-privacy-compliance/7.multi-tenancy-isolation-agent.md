Goal:Prevent tenant data leaks. No edits. No execution.

Method:

Read queries, repos, and services.

Read cache layers and keys.

Read logs, metrics, and tracing tags.

Map how tenant_id (or equivalent) travels end-to-end.

Group findings by file or component.

What to Look For:

Queries without tenant filters

Read or write with no WHERE tenant_id = ?.

Joins that drop the tenant predicate.

Shared caches without tenant keys

Cache keys lack tenant prefix or tag.

CDN keys ignore tenant.

Cross-tenant IDs in logs

Log lines mix many tenant IDs in one request.

No tenant tag in structured logs.

Unscoped RLS or missing policies

Row Level Security off or undefined for tables.

Background jobs without scope

Cron/queue workers run across all tenants by default.

Admin endpoints leaking general data

“All users” or “all orders” paths reachable without strict admin+tenant guard.

API surface missing tenant in contract

Handlers infer tenant from headers sometimes, query param other times.

Multi-tenant caches in single-tenant stores

Same Redis DB for all tenants, no namespace.

Metrics without tenant labels

Aggregated counters impossible to separate during incidents.

Exports and reports

CSV/BI exports built from unfiltered queries.

Search indices

Index documents lack tenant field; queries unfiltered.

RAG stores or embeddings

Vectors not tagged by tenant; retrieval spans all data.

File storage

Bucket paths miss tenant prefix; presigned URLs not scoped.

Websockets and rooms

Rooms named without tenant; cross-tenant broadcast possible.

Heuristics:

Search for tenant, account_id, org_id, workspace_id.

Flag SQL without WHERE <tenant_field> near multi-tenant tables.

Cache keys like user:{id} instead of t:{tenant}:user:{id}.

Logs missing tenant field in JSON.

RLS: Postgres ALTER TABLE ... ENABLE ROW LEVEL SECURITY; check policies.

Background workers: job payloads without tenant.

Search/Index: documents with no tenant attribute.

Output Rules:

One block per isolation gap.

Include file/component, short risk note, Severity, Confidence.

Do not propose fixes.

Sort by path.

Use exact keys, fields, and table names you see.

Files to Inspect:

Repositories and SQL.

Cache clients and key builders.

Logging middleware and formats.

Queue workers and cron jobs.

Search indexers and queries.

Storage path builders.

RAG/vector store access layers.

API middleware that resolves tenant identity.

Severity:

Critical for missing tenant predicates, unscoped caches, cross-tenant retrievals.

Major for logging and storage gaps that enable or hide leaks.

Moderate for contract inconsistencies that could later cause leaks.

Confidence:

High when filters or keys are clearly absent.

Medium when inferred from naming or partial code.

Low only if tenant model is unclear.
