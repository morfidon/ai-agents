Goal:Know where sensitive data flows. No edits. No execution.

Method:

Read data models, schemas, ETL scripts, and analytics pipelines.

Read field-level lineage maps or metadata stores (OpenLineage, Amundsen, DataHub, Marquez, etc.).

Trace personally identifiable information (PII) from origin to sink.

Group findings by table, service, or pipeline.

What to Look For:

PII fields without lineage records

Columns like email, phone, ssn, user_id, address missing lineage annotations.

No mapping in data catalog or lineage metadata.

Unmasked exports

CSV or Parquet exports contain raw PII.

External systems receive unmasked user data.

Analytics sinks with raw data

BI or ML datasets include sensitive columns not anonymized.

Warehouse tables mixing PII and behavioral metrics.

No data classification tags

Schemas lack labels like pii=true or sensitivity=high.

ETL code doesn’t propagate classification tags downstream.

Inconsistent field names

Same PII field labeled differently across systems (customer_id vs userId vs uid), making lineage tracking unreliable.

Untracked derived columns

PII-derived hashes or tokens without trace back to source field.

No record of how identifiers are obfuscated or salted.

Cross-environment leaks

Production data copied into staging or dev without masking.

No masking in data sharing

Data shared with partners or models includes raw user identifiers.

Unlogged ETL data movements

Batch scripts or direct SQL copies without lineage registration.

Analytics dashboards exposing PII

Frontend visualizations display names, emails, or IDs.

RAG or vector stores embedding PII text

Embeddings contain personal data used for semantic search.

Inconsistent key rotation for PII encryption

Encrypted fields tied to old or inactive KMS keys.

GDPR data export & delete flows incomplete

No link from data subject to stored records in multiple systems.

No DLP or scanning coverage

Missing automated scans to detect PII fields in storage and codebases.

Heuristics:

Search schema names for email, ssn, address, phone, id, token, dob.

Check metadata YAMLs, lineage JSONs, catalog tags.

Inspect ETL/SQL for SELECT * on tables containing PII.

Look for exports to S3/GCS with no _masked suffix or anonymization functions.

Cross-reference with DLP or policy lists.

Check logs or embedding scripts for full-text PII.

Output Rules:

One block per lineage or PII gap.

Include dataset, pipeline, or file, short note, Severity, Confidence.

No remediation advice.

Sort by system or data path.

Use exact resource names and column identifiers you see.

Files to Inspect:

Data warehouse schemas.

ETL jobs, Airflow DAGs, dbt models.

Metadata and lineage files (.yaml, .json, .sql).

Export and analytics scripts.

Vector or embedding indexes.

Environment configs (dev/staging/prod).

Severity:

Critical for unmasked exports, cross-environment leaks, or analytics using raw PII.

Major for missing lineage, no tags, or untracked derived PII.

Moderate for inconsistent naming or missing documentation only.

Confidence:

High when explicit fields or patterns detected.

Medium when inferred from partial lineage or naming.

Low only if data dictionary incomplete.
